**Этап 2 – Анализ требований и первый User Story Mapping**

**Цель этапа:** разбить проект на функциональные модули, сформулировать пользовательские истории, построить карту пользовательских сценариев, найти визуальные ориентиры, заложить структуру данных и создать бэклог для планирования разработки.

**1\. Системный аналитик (Бондаренко) – User Stories и Use Case диаграмма**

1.1. Подробные пользовательские истории с критериями приёмки

Все истории приоритизированы методом MoSCoW:

**Must have** – без этого продукт не работает.

**Should have** – важные, но не критические для первого релиза.

**Could have** – желательные, можно отложить.

|     |     |     |     |
| --- | --- | --- | --- |
| ID  | Что хочет пользователь | Приоритет | Критерии приемлемости |
| US-01 | Как новый пользователь, я хочу зарегистрироваться по email и паролю, чтобы иметь личный аккаунт | Must | **1.** Пользователь вводит email и пароль (мин. 6 символов). **2.** Если email не занят, создаётся учётная запись, возвращается 201. **3.** Если email уже существует, возвращается 409 с сообщением «Пользователь с таким email уже существует». **4.** Пароль хешируется, в ответе не возвращается. |
| US-02 | Как зарегистрированный пользователь, я хочу войти в систему, чтобы получить доступ к своим задачам | Must | **1.** Пользователь вводит email и пароль. **2.** При совпадении возвращается JWT-токен. **3.** При неверных данных возвращается 401 «Неверный email или пароль» (без уточнения, что именно неверно). **4.** Токен действителен 24 часа. |
| US-03 | Как авторизованный пользователь, я хочу создать задачу с названием, описанием, сроком и приоритетом, чтобы зафиксировать то, что нужно сделать | Must | **1.** Принимаются поля: title (обязательно), description (опционально), due_date (опционально, валидная дата в будущем), priority (low/medium/high, по умолчанию medium). **2.** Задача связывается с пользователем. **3.** Возвращается созданная задача с id, статусом «active» и временем создания. **4.** Без авторизации – 401. |
| US-04 | Как пользователь, я хочу видеть список своих задач, чтобы оценить текущую нагрузку | Must | **1.** По GET-запросу возвращаются только задачи текущего пользователя. **2.** Поддержка фильтрации по проекту, статусу, приоритету через query-параметры. **3.** По умолчанию сортировка по дате создания (сначала новые). |
| US-05 | Как пользователь, я хочу редактировать задачу, чтобы уточнить детали | Must | **1.** Можно изменить любое поле, кроме id и user_id. **2.** PUT или PATCH запрос обновляет задачу. **3.** Нельзя редактировать чужую задачу (403). |
| US-06 | Как пользователь, я хочу отметить задачу выполненной (чекбокс), чтобы видеть прогресс | Must | **1.** При изменении статуса на «completed» задача остаётся в списке, но отображается перечёркнутой. **2.** Можно вернуть статус обратно на «active». |
| US-07 | Как пользователь, я хочу удалить задачу, если она больше не актуальна | Should | **1.** DELETE запрос удаляет задачу только её владельца. **2.** Задача удаляется из БД (без возможности восстановления в MVP). |
| US-08 | Как пользователь, я хочу создавать проекты (категории), чтобы группировать задачи | Should | **1.** Проект имеет название, принадлежит пользователю. **2.** Задаче можно указать project_id (необязательно). **3.** При удалении проекта связанные задачи не удаляются, а теряют привязку (project_id = NULL). |
| US-09 | Как пользователь, я хочу фильтровать задачи по проекту, статусу и приоритету, чтобы быстро находить нужное | Should | **1.** Фильтр по проекту: GET /tasks?project_id=1. **2.** Фильтр по статусу: GET /tasks?status=active. **3.** Фильтр по приоритету: GET /tasks?priority=high. **4.** Можно комбинировать. |
| US-10 | Как пользователь, я хочу получать напоминание о приближающемся сроке задачи (email/push) | Could | В первой версии не реализуется, только закладывается поле due_date для будущего использования. |

**2\. UI/UX Дизайнер (Провоторова) – Референсы и первый набросок интерфейса**

2.1. Анализ конкурентов и референсов

|     |     |     |
| --- | --- | --- |
| Сервис | Ключевые UI-решения | Что берём в проект |
| **Todoist** | Быстрое добавление задачи через «+» и строка ввода вверху списка, цветные метки приоритета, группировка по проектам в левом сайдбаре. | Строка быстрого добавления задачи вверху главного экрана. Цветные индикаторы приоритета (красный – high, жёлтый – medium, серый – low). |
| **Microsoft To Do** | Светлая минималистичная тема, список «Мой день», анимация завершения задачи (зачёркивание с плавным движением). | Минималистичный дизайн в светлых тонах, чекбокс с зачёркиванием выполненной задачи. |
| **Trello** | Карточки с обложками, drag-and-drop между колонками (если делать канбан). | Не берём канбан в первом релизе, но используем идею цветовых меток проектов. |
| **Notion** | Гибкие представления: список, доска, календарь. | Оставляем только списочное представление на MVP, с возможностью будущего расширения. |

**3\. Backend-разработчик (Плеханов) – Набросок структуры БД и техническая реализуемость**

User (id, email, password_hash, created_at)

Project (id, name, user_id(FK), created_at)

Task (id, title, description, due_date, priority(enum), status(enum), project_id(FK, nullable), user_id(FK), created_at, updated_at)

Пока минимально – без хранения уведомлений, историю изменений и т.п.
